# Canonical Architecture

## Status

STATUS: CANONICAL ARCHITECTURE AUTHORITY

## 1. Purpose

The Truth-Seeking Interoperability Constructor Factory builds compliance determinations by preserving distinctions across source contracts, live evidence, adapter promotion records, export projections, and client delivery boundaries. It does not allow a presentation layer or remediation helper to invent truth outside the promoted factory chain.

## 2. Layer Model

| Layer | Name | Role |
| --- | --- | --- |
| L0 | Source inputs | Raw STIG data, assertion contracts, UI templates, and reference corpora |
| L1 | Kernel / coalgebra truth engine | General construction, witness, replay, and ledger semantics |
| L2 | Catalog / STIG contract semantics | STIG contract DSL, lawful partitions, measurable bindings |
| L3 | Live adapter family promotion | Runtime observation bridges promoted by fixtures, replay, and DP gates |
| L4 | Live break/fix regression | Live campaign evidence, real open findings, external evidence blockers |
| L5 | Export projection / web_app | Governed rendering and live surface constrained by promoted artifacts |
| L6 | Client deliverability | Supported-only client bundle and declared support boundaries |
| L7 | Backlog / noncanonical notes | Future work, superseded patch plans, unresolved evidence |

## 3. Domain Coalgebras

A domain coalgebra must define:

- carrier objects: the systems, controls, or observations under review
- observables: fields that can be measured from approved evidence sources
- transition/event structure: how observations are captured, replayed, and compared
- lawful partitions: pass/open/unresolved classifications that preserve source distinctions
- witnesses: records that prove why a determination was made

STIG domain semantics are carried by `assertion_contracts.json`, `disa_stigs.json`, distinction-preserving tests, and generated catalog records.

## 4. Maturity Metric Coalgebra

Maturity is measured by hard gates and metric records, not by prose claims. Required invariant metrics include:

- replay fidelity equals 1.0 for promoted paths
- distinction loss rate equals 0
- projection equivalence rate equals 1.0
- unresolved preservation rate equals 1.0
- frontend truth invention incidents equals 0
- role drift incidents equals 0

Partial maturity may be reported only with explicit production gaps and waste accounting.

## 5. Adapter Family Promotion Coalgebra

Adapter families are observation bridges. They may become live-supported only after promotion evidence proves:

- required evidence fields are bound to runtime paths
- good, bad, malformed, absent, noisy, boundary, disabled, and out-of-scope fixtures behave lawfully
- distinction-preserving gates pass
- replay is deterministic
- live break/fix evidence does not contradict the promoted family

The promoted artifact is `LiveAdapterPromotionPortfolio`. No family may be skipped or assumed promoted.

Current bridge evidence promotes 60 of 67 controls. Seven controls remain unresolved because they require external evidence or represent real open findings.

## 6. Export Projection Coalgebra

Export is a governed projection, not a truth engine. It may render only certified fields from factory artifacts.

The export/web_app layer must not:

- parse STIG criteria DSL as an independent evaluator
- compare live evidence to requirements outside the promoted adapter path
- relabel unresolved controls as resolved
- show live validation for unsupported controls
- widen product support claims beyond gate records

The export may provide a live tool surface only when the live path calls promoted backend/adapters and preserves the canonical status vocabulary. A static projection wrapper is not a replacement for the live tool; it is only a certified read-only delivery surface.

## 7. Delivery Readiness Coalgebra

Client delivery packages a scoped claim:

- what is live-supported
- what is factory-validated only
- what requires external evidence
- what requires manual attestation
- what is out of scope
- what is redesign-required

The client bundle is deliverable only if the `ClientDeliverabilityGateRecord` passes and the bundle declares its support boundary.

## 8. Forbidden Role Drift

Core rule:

```text
Factory = truth engine
Adapter = observation bridge
Export = governed projection
Client bundle = scoped deliverable
```

Forbidden drift:

- export becoming an evaluator
- UI becoming an adjudicator
- remediation advice becoming evidence
- source notes becoming canonical status
- client packaging becoming a full-live product claim without gates

## 9. Status Vocabulary

Canonical statuses:

- `PROMOTED`: adapter/control passed required promotion gates
- `DEMOTED`: previously promoted path failed regression or gate evidence
- `PROJECTED_UNRESOLVED`: export preserves unresolved status from non-promoted path
- `INSUFFICIENT_EVIDENCE`: evidence exists but is not enough for a lawful determination
- `BLOCKED_EXTERNAL`: requires organization-provided evidence outside appliance capture
- `REDESIGN_REQUIRED`: current model cannot lawfully represent the requirement
- `DELIVERABLE_SUPPORTED_ONLY`: client bundle supports only promoted controls/families
- `DELIVERABLE_WITH_DECLARED_BOUNDARIES`: client bundle is deliverable with explicit unsupported/blocked scope

## 10. Client Claim Boundaries

The current acceptable claim is supported-only delivery: 60 live-resolved controls, 5 blocked-external controls, and 2 open findings in the current bridge projection. Older campaign docs mention 6 blocked-external; the latest passing bridge/export gate artifacts control the release posture until superseded by a new gate run.

Do not use the phrase "full live STIG product" unless all required live and external evidence gates pass.
